# Command: `/waves:rules-update`

**Status:** ✅ DESIGNED

---

## Overview

**Purpose:** Detect code changes since last rules update and propose new rules, modifications, or deprecations.

**Input:** Existing rules file + `updated_at` timestamps on individual rules

**Output:** Updated rules file with new/modified/deprecated rules

**Parameters:** `[layer]` (optional) - Specific layer to re-analyze

**Flow Derivations:**
- **A:** Software (Git-based) → Detect changed files → Re-analyze affected layers → Propose updates
- **B:** Software (Timestamp-based) → Same as A but using file dates
- **C:** General → User-guided updates (conversational)

---

## Key Differences from `rules-create`

| Aspect | rules-create | rules-update |
|--------|--------------|--------------|
| Starting point | No rules exist | Rules already exist |
| Analysis scope | Full layer analysis | Only changed files |
| Output | New rules file | Updated rules file |
| Rule IDs | Start at 1 | Continue from max ID |
| Deprecation | N/A | Can deprecate obsolete rules |

---

**═══════════════════════════════════════════════════════════════════**
**FLUJO A: SOFTWARE (Git-based)**
**═══════════════════════════════════════════════════════════════════**

```
STEP 0: Prerequisites
        └→ Check project_rules.json exists
        └→ Read last rule updated_at timestamp

STEP 1: Detect changes (reuse manifest-update logic)
        └→ git diff since oldest rule updated_at
        └→ Filter to code files only (no autogenerated)
        └→ Categorize: Modified / New / Deleted

STEP 2: Map changes to layers
        └→ src/components/* → presentation_layer
        └→ src/services/* → data_layer
        └→ src/api/* → api_layer
        └→ (Use manifest architecture_patterns_by_layer)

STEP 3: Re-analyze ONLY affected layers
        └→ Invoke same subagents as rules-create
        └→ But scoped to changed files

STEP 4: Compare with existing rules
        ├─ New patterns found? → Propose NEW rules
        ├─ Existing pattern changed? → Propose MODIFICATION
        └─ Pattern no longer exists? → Propose DEPRECATION

STEP 5: Show proposals
        ┌─────────────────────────────────────────┐
        │ ➕ NEW RULES (3)                        │
        │ • [presentation_layer] New hook pattern │
        │                                         │
        │ ✏️ MODIFICATIONS (1)                    │
        │ • Rule #5: Update description           │
        │                                         │
        │ ⚠️ DEPRECATIONS (1)                     │
        │ • Rule #3: Pattern no longer in use     │
        │                                         │
        │ ⚠️ NEW ANTIPATTERNS (2)                 │
        │ • Props drilling in new component       │
        └─────────────────────────────────────────┘

STEP 6: User confirms → Apply updates
        └→ Add new rules with new IDs
        └→ Update modified rules (set updated_at)
        └→ Mark deprecated rules (or remove)
```

## Subagents (Reuse from rules-create)

- **convention-detector** (scoped to changed files)
- **antipattern-detector** (scoped to changed files)
- **rule-comparator** (NEW: compares existing vs detected)
